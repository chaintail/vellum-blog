why this course

The gap: everyone teaches building. Nobody teaches reading, decoding, and tracing.

Existing Solana education (Rareskills, School of Solana, official bootcamps) is aimed at writing your first program. Almost nothing teaches developers to read Solana: decode raw transactions, trace CPIs and token flows, audit account validation, reverse engineer unknown programs. That literacy is exactly what an experienced EVM developer needs to become dangerous on Solana quickly, and it's what this course delivers.

Instead of generic blockchain basics, every lesson isolates one Solana concept and compares it against the closest Ethereum mental model. One idea at a time: small enough to be a slide sequence, a livestream segment, and a short-form video.

The format is the distribution hook. This curriculum is the actual roadmap we designed to train ourselves. The series is two EVM-native devs genuinely learning Solana on camera, and the real "wait, token balances don't live in the contract?" moments are relatable to exactly the audience Solana wants to convert.

Goal · Stop thinking "contract with storage." Start thinking in accounts, programs, instructions, and explicit account access.
01

What a Solana address is

Solana addresses as base58-encoded 32-byte public keys, then immediately break the Ethereum habit of reading an address as "EOA vs contract."

  • Compare 0x... Ethereum addresses with Solana base58 public keys
  • Show that wallets, programs, token mints, token accounts, and PDAs can all look like the same "address" shape
  • Introduce the habit: fetch account metadata before assuming what an address is
base58public keysEOA reset
02

SOL, lamports, and balances

SOL as the native asset and lamports as the smallest unit, connected to fees, rent exemption, and account creation.

  • Define 1 SOL = 1,000,000,000 lamports
  • Explain why non-wallet state accounts hold SOL for rent exemption
  • Contrast Ethereum gas payment with Solana's smaller base fees and account-funding requirements
lamportsfeesrent exemption
03

What a transaction is on Solana

A Solana transaction as a signed message containing one or more instructions, a recent blockhash, and an explicit account list.

  • Explain signatures, signer ordering, message headers, account keys, and recentBlockhash expiration
  • Compare a Solana multi-instruction transaction to an Ethereum transaction that usually targets one contract call
  • Preview why account lists are the key to transaction decoding later
signaturesrecent blockhashinstructions
04

Programs vs accounts

The most important architectural shift: program code and state do not live together the way contract code and storage do on Ethereum.

  • Programs are executable accounts
  • State lives in separate accounts owned by a program
  • Protocol design becomes: which state accounts exist, who owns them, and which instructions can mutate them
programsstate accountsownership
05

Read-only vs writable accounts

Why every instruction declares the accounts it reads and writes, and how this drives Solana's parallel execution model.

  • Define account metas: signer/non-signer and writable/read-only
  • Show why writable accounts create locks and can bottleneck throughput
  • Introduce the design mantra: avoid unnecessary hot writable accounts
account metaswrite locksparallelism

Instructions

The basic execution unit: a program id, account metadata, and opaque instruction data bytes.

  • Map programId, accounts, and data to what an explorer or RPC response shows
  • Explain why instruction data is meaningless until decoded with a schema or IDL
  • Connect instructions to later topics like Anchor discriminators and compiled instruction indexes

Cross-program invocation (CPI)

Solana's version of one program calling another, but with explicit accounts and privilege rules.

  • Compare CPI to an EVM external call, then highlight where the analogy breaks
  • Show how CPIs appear as inner instructions in transaction traces
  • Explain signer/writable privilege propagation and why CPI is central to security review
CPIinner instructionsprivileges

Fees, compute units, and priority fees

Compute units vs EVM gas, compared carefully without pretending they are the same economic model.

  • Introduce compute unit limits and the ComputeBudget program
  • Explain prioritization fees and why compute budget instructions often appear at the start of transactions
  • Discuss how simulation helps estimate compute before sending a transaction
compute unitspriority feesComputeBudget
09

Atomicity and rollback behavior

Transaction-level atomicity: if one instruction fails, the whole transaction fails and state changes roll back.

  • Show how multi-instruction transactions either fully commit or fully fail
  • Explain what logs can still reveal after a failed transaction
  • Compare this to Ethereum revert behavior while noting Solana's explicit account model
atomicityrollbackfailed tx

Program logs and return-style debugging

Logs, simulation output, and custom errors as the first tools for understanding what happened inside a transaction.

  • Recognize Program log:, Anchor errors, and custom program error codes
  • Use logs to map execution order and CPI boundaries
  • Preview a practical trace workflow: simulate, inspect logs, decode accounts, then map to code

Associated token accounts

Why token balances live in token accounts instead of inside wallets or token contracts.

How protocol state is usually partitioned

The common account layout patterns used by Solana protocols.

Concurrency-aware design

How Solana architecture changes when every writable account can become a throughput bottleneck.

  • Avoid global mutable counters or single hot state accounts where possible
  • Shard state by user, market, epoch, pool, or position
  • Design instruction paths so unrelated users can execute in parallel
write locksshardingthroughput

Solana-specific security mindset

The core Solana threat model: users submit account lists, and programs must verify every account that matters.

Reinitialization and duplicate initialization bugs

Bugs that happen when programs fail to distinguish fresh accounts from previously initialized accounts.

  • Check discriminators, version fields, initialization flags, and expected zeroed state
  • Discuss close-and-reopen edge cases and stale data
  • Compare Anchor init, init_if_needed, and manual initialization patterns

Precision and math issues

Financial math hazards that become protocol-breaking when balances, shares, prices, and decimals interact.

  • Study integer division, rounding direction, decimal mismatches, overflow assumptions, and share accounting
  • Track invariants like total assets vs total shares
  • Show why a "small rounding issue" can become a drain vector

Denial-of-service and lock contention

Non-theft failures where an attacker or bad architecture prevents useful transactions from landing.

  • Design mitigations: sharding, limits, preflight checks, and bounded loops

  • Teach users how to read upgrade status as part of protocol due diligence

production model

One lesson, five artifacts

The curriculum is structured to become a repeatable content pipeline. Tone: practical, developer-native, conversational. The recurring narrative: "I already know Ethereum. What do I need to unlearn, relearn, and practice to become effective on Solana?"

Learn the micro-lesson

The host works through one focused lesson conversationally until the concept is clear enough to teach.

Convert to slides

A 3–5 slide mini-deck: one core idea, one Ethereum comparison, one Solana example, one gotcha.

Teach it live

Host teaches a colleague who is also learning Solana. Authentic peer-learning livestreams, held as X Spaces cohosted from @VellumNetwork @ClanWorldGame @ChaintailLabs @Mikail_xyz.

Edit into shorts

The best explanation is cut into a short optimized for X, TikTok, Instagram Reels, and YouTube Shorts.

Publish the series

Each lesson stands alone; the full set becomes a structured Solana onboarding course for EVM developers, published under Vellum Network.

proposed grant scope

Phase 1: foundations to transaction literacy

the ask

31 lessons · 31 shorts · 6 livestreams · $10,000

Take a developer from zero Solana knowledge to independently reading and decoding real transactions from raw JSON. Estimated 105–165 combined team hours across research, slides, livestreams, and short-form editing. Roughly 60% of the course for about 45% of the total effort: the deep-research exploit and architecture content is deliberately held for Phase 2.

Phase 1 committed deliverables
  • 31 micro-lesson outlines (themes 1–6)
  • 31 short-form videos
  • 6 livestreams or recorded teaching sessions, one per theme
  • 31 mini slide decks (3–5 slides each)
  • Open-source curriculum repo (CC-BY)
  • Public retrospective with real metrics
M1 · 30%

Format pilot · ~3 weeks

Lessons 1–5 fully shipped: slide decks, one livestream, five published shorts, plus the reusable slide template and editing workflow. If the format doesn't work, everyone finds out cheaply.

M2 · 40%

Core execution · ~5 weeks

Lessons 6–20 shipped (execution, state layout, standard programs) at a ~3 lesson/week cadence, one livestream per theme.

M3 · 30%

Program & transaction literacy · ~3 weeks

Lessons 21–31 shipped (Rust/Anchor reading, transaction decoding). All 31 shorts published, slide decks open-sourced (CC-BY) with notes and helper scripts where useful, plus a public retrospective with real metrics.

Phase 2 (follow-on application): themes 7–10. Protocol architecture, security patterns, historical exploit labs, and reverse-engineering capstones. The deeper, more expensive arc, and arguably a standalone security-education series. We'd rather earn it with a shipped Phase 1 and real audience data than promise all 49 lessons upfront.
